# CACE characterization — saradc comparator

Datasheet-driven characterization of the StrongARM comparator with pass/fail vs. spec,
across PVT corners. Built from the same testbenches in `../tb/`.

## Layout
```
saradc/
├── comparator.sch / .sym          DUT (CACE netlists this into cace/netlist/)
├── cace/
│   ├── comparator.yaml            datasheet: parameters, specs, conditions
│   ├── templates/
│   │   ├── xschemrc               sources the GF180 PDK xschemrc + adds saradc/ to libpath
│   │   ├── tran.sch               regen_time  (meas -> echo $&tregen)
│   │   └── kickback.sch           kickback    (meas -> echo $&kick)
│   ├── netlist/  doc/  runs/      auto-generated by CACE
└── tb/                            the interactive design testbenches
```

## Install (in the IIC-OSIC-TOOLS container)
```
python3 -m pip install --upgrade cace        # or use the Nix flow from the CACE docs
export PDK_ROOT=$PDK_ROOT                      # must point at the dir containing gf180mcuD
export PDK=gf180mcuD
```

## Run (from the `saradc/` directory)
```
cace cace/comparator.yaml                                  # all parameters, best netlist
cace cace/comparator.yaml --source schematic              # force schematic netlist
cace cace/comparator.yaml --parameter regen_time \
     --source schematic --log-level DEBUG \
     --parallel-parameters 1 --sequential                 # debug a single parameter
```
CACE generates one netlist per condition combination (e.g. regen_time = 3 corners × 3 temps
= 9 sims), runs ngspice, reads each `*.data`, and prints a pass/fail summary vs. the specs.

## How a template maps to a result
- `template:` is the schematic in `templates/`. It instances `comparator.sym`, includes the
  DUT via `CACE{DUT_path}`, the PDK via `CACE{PDK_ROOT}/CACE{PDK}/...`, and substitutes
  conditions (`CACE{vdd}`, `CACE{vcm}`, `CACE{corner}`, `CACE{temperature}`, expressions in
  `CACE[...]`).
- A scalar result is emitted with `echo "$&var" > CACE{simpath}/CACE{filename}_CACE{N}.data`;
  the datasheet's `variables:` list names the column(s). (For waveforms use `wrdata` instead.)

## GF180 specifics to confirm on first run
- **Model path**: templates use `$PDK_ROOT/gf180mcuD/libs.tech/ngspice/{design.ngspice, sm141064.ngspice}`
  — this is the `$::180MCU_MODELS` your xschem testbenches use. Confirm the path in the container.
- **Corner names** are the `sm141064.ngspice` section names: `typical`, `ss`, `ff` (add `sf`,`fs`).
  GF180 uses `typical`, **not** `tt`.

## Likely first-run gotchas (can't be verified from the host)
1. **DUT double-definition** — if ngspice errors that subckt `comparator` is redefined, the
   template netlist is both instancing the symbol *and* expanding its schematic. Fix by making
   CACE treat the DUT as a black box (it normally does via `CACE{DUT_path}`); check the
   `--source` netlist and the symbol's netlisting attributes.
2. **`meas ... when` fails** if the comparator doesn't resolve in a bad corner — that's a real
   fail, but verify the 12 ns window / thresholds first.
3. **`code_shown.sym`/`lab_pin.sym` not found** — the `templates/xschemrc` must source the PDK
   xschemrc successfully (needs `PDK_ROOT`/`PDK` exported).

## Next parameters (templates to add)
- `offset` — input-referred offset; for random σ needs the GF180 **mismatch/Monte-Carlo** models
  (CACE supports MC via `.option SEED` + `collate`). Nominal offset: ramp vid, meas the flip.
- `cm_range` — sweep `vcm` as a CACE condition; emit decision margin per VCM (a plot parameter).
- `noise` — ngspice has no device transient noise; rely on eq.15 (γ from STH), or inject the
  σ via `trnoise` for decision-error statistics (see `../tb/comparator_tb_noise.sch`).
